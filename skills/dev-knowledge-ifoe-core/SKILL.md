# Core IFoE

**IFoE** (Infinity Fabric over Ethernet; also known as **UALoE**) is a way to
transport **SDP** (the SoC-15 Scalable Data Port — see the
`dev-knowledge-sdp-interface` skill) over an **ethernet fabric**.

## Where the core block sits

The core IFoE block has:

- **Top edge** — two SDP interfaces: an **Originator** SDP interface and a
  **Completer** SDP interface. (The Originator interface is also referred to as the
  **initiator** interface.)
- **Bottom edge** — pairs to **ethernet hardware**.

It takes SDP traffic arriving on its top-edge SDP interfaces and moves it across
ethernet to a peer IFoE block, which presents it back out on SDP at the far end.

## Streams

Core IFoE manages SDP traffic as **streams**. A stream is identified by the
combination of:

| Field                    | Width   | Meaning                                          |
|--------------------------|---------|--------------------------------------------------|
| Destination accelerator id | —     | which accelerator the stream targets             |
| Path                     | 3 bits  | generated by **hashing the address**             |
| Host vs guest            | 1 bit   | host or guest traffic                            |
| Request vs response      | 1 bit   | whether the stream carries requests or responses |

The path being an address hash spreads traffic for a given destination across up
to eight paths; `(accelerator id, path)` is also the key used downstream for
port routing (see `dev-knowledge-ifoe-subsystem`).

## TCP-like per-stream transport

Each stream is **TCP-like**: it has a **sequence number**, **acknowledgements**,
and a **replay buffer** (so lost/dropped traffic can be retransmitted from the
replay buffer).

## What each stream carries

- **Request stream** carries **`Req`** and **`OrigData`** (the Originator-produced
  SDP message types).
- **Response stream** carries **`RdRsp`** and **`WrRsp`** (the Completer-produced
  SDP message types).

(See `dev-knowledge-sdp-interface` for what those message types are.)

## Acknowledgement aggregation

The acknowledgements for a direction's traffic travel back on the **response
stream of the opposite direction**. Concretely:

> The **B→A response stream** carries the ACKs for **both** the **A→B request
> stream** and the **A→B response stream**.

So a response stream does double duty: it carries `RdRsp`/`WrRsp` *and* the
TCP-style ACKs for the two opposite-direction streams.
